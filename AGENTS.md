## Repository Overview

Kessel SDK Browser is an NX monorepo providing browser SDKs for Project Kessel -- Red Hat's fine-grained authorization system. The SDK lets frontend applications check user permissions for UI rendering decisions (show/hide, enable/disable).

Published package: `@project-kessel/react-kessel-access-check`

Key domain concepts:
- **Access checks** verify whether the current user has a `relation` (view, edit, delete) to a `resource` (workspace, group)
- **Resources** have `id`, `type`, and `reporter` fields. The `reporter` is typically `{ type: 'rbac' }`
- **Bulk checks** send multiple resource/relation pairs in a single request
- Frontend checks are UX-only -- the backend always re-validates permissions

## Architecture

The SDK has a strict two-layer architecture:

```
React layer (src/)           Core layer (src/core/)
  AccessCheckProvider.tsx      api-client.ts
  AccessCheckContext.tsx       transformers.ts
  hooks.ts                     workspace-client.ts
  types.ts
  index.ts
```

Dependency direction is one-way: React layer imports from core, never the reverse. The core layer has no React dependencies -- pure and async-pure functions only.

The React layer exposes a Provider/Context pattern:
```
<AccessCheck.Provider> --> React Context --> useSelfAccessCheck() hook
```

The `useSelfAccessCheck` hook has three TypeScript overloads (single resource, bulk same-relation, bulk nested-relations). Both internal hooks are always called unconditionally to satisfy Rules of Hooks -- an `enabled` flag controls which performs work.

## Naming and Code Style Conventions

### Files

- React components: `PascalCase.tsx` (e.g., `AccessCheckProvider.tsx`)
- Hooks: `hooks.ts` (single file for all hooks in this package)
- Pure utilities: `kebab-case.ts` (e.g., `api-client.ts`, `workspace-client.ts`)
- Tests: colocated as `<source-file>.test.tsx` or `<source-file>.test.ts`
- Integration tests: `__tests__/integration/<concern>.test.tsx`

### TypeScript

- Strict mode enabled with `noUnusedLocals` and `noImplicitReturns`
- All shared types live in `types.ts` -- no inline type definitions in components or hooks
- Unused function parameters: prefix with `_` (ESLint rule `argsIgnorePattern: '^_'`)
- `no-explicit-any`: warning in production code, disabled in test files
- Function overloads: use `@typescript-eslint/no-redeclare` (base `no-redeclare` is off)

### Exports

Public API is intentionally minimal. Only export what consumers need from `index.ts`. Internal helpers (`checkSelf`, `checkSelfBulk`, transformers, `ApiConfig`) stay unexported. When adding a public export, update `src/index.ts`.

### Commits

Conventional commits enforced by commitlint. Format: `<type>(<scope>): <subject>`

Required scopes: `react-kessel-access-check`, `demo-react-kessel-access-check`, `release`

Valid types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`, `revert`, `ci`, `build`

Version impact: `feat` = minor bump, `fix` = patch bump, `BREAKING CHANGE:` or `!` suffix = major bump. Other types do not trigger a release.

## Testing Patterns and Infrastructure

### Two Mocking Strategies

The codebase uses two distinct mocking approaches. Use the correct one for each test type:

**MSW (Mock Service Worker)**: Used in integration tests and some unit tests (`api-client.test.ts`, `hooks.bulkConfig.test.tsx`). Intercepts `fetch` at the network level.
```typescript
import { server } from '../api-mocks/msw-server';
beforeAll(() => server.listen({ onUnhandledRequest: 'error' }));
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

**jest.fn() on global.fetch**: Used in unit tests (`hooks.test.tsx`, `AccessCheckProvider.test.tsx`, `workspace-client.test.ts`). Provides fine-grained control over response shapes.

### Test Utilities (api-mocks/test-utils.tsx)

- `createTestWrapper(config?)` -- Provider wrapper for `renderHook`
- `createMockResource(overrides?)` -- single resource factory
- `createMockResources(count, overrides?)` -- batch resource factory
- `createMaliciousResource(type)` -- XSS/injection payloads for security tests
- `expectValidErrorStructure(error, includeDetails?)` -- asserts error shape

### Jest Configurations

Three jest configs exist for `react-kessel-access-check`:
- `jest.config.js` -- runs all tests (NX default target)
- `jest.config.unit.js` -- unit tests only
- `jest.config.integration.js` -- integration tests only

CI runs unit and integration separately. The `npm test` shorthand runs all.

### Environment Setup

`setupTests.ts` polyfills `TextEncoder`, `TextDecoder`, `ReadableStream`, `TransformStream`, `WritableStream`, `BroadcastChannel`, and `cross-fetch` for jsdom. MSW v2 requires these.

## Common Pitfalls

1. **Forgetting to update all three hook overloads.** Any change to `useSelfAccessCheck` must update all three overload signatures, the implementation signature, and tests for each variant.

2. **Mixing up the two error levels.** Top-level `result.error` is for HTTP/network failures. Per-item `result.data[i].error` is for individual check failures in bulk responses. Always check both.

3. **Conditional hook calls.** Both internal hooks (`useSingleAccessCheck`, `useBulkAccessCheck`) must always be called unconditionally. The `enabled` flag controls which one runs. Do not wrap hook calls in `if` statements.

4. **Breaking the dependency direction.** `core/` must never import from the React layer or from React itself. If you need a type in `core/`, it must come from `../types`.

5. **Test file in build output.** Test files, `api-mocks/`, and `__tests__/` are excluded from the build via `tsconfig.lib.json`. If you add new test directories, exclude them there.

6. **Missing commit scope.** Commitlint requires a scope. Omitting it (e.g., `feat: add feature`) will fail CI. Use one of the three allowed scopes.

7. **Empty arrays in bulk checks.** The `resources` param uses `NotEmptyArray<T>` (`[T, ...T[]]`). In tests, cast empty arrays with `as any` to test edge cases. Production code should not pass empty arrays.

8. **AbortController signal not wired.** The cleanup pattern creates an `AbortController` but the signal is not currently connected to `fetch`. The `isMounted` guard is what prevents post-unmount state updates.

9. **Workspace client uses a different API.** `fetchRootWorkspace` and `fetchDefaultWorkspace` hit the RBAC v2 API (`/api/rbac/v2/workspaces/`), not the Kessel access check endpoints. The base URL may differ.

## PR Expectations

### CI Must Pass

All PRs must pass: lint, unit tests, integration tests, build, and build artifact verification. Commitlint validates every commit message in the PR against the conventional commit format.

### What Reviewers Check

- Public API changes are intentional and documented (update `index.ts`, README, types)
- All three hook overloads are tested when touching `useSelfAccessCheck`
- Core layer remains React-free
- Error handling covers both top-level and per-item errors
- New test files are excluded from `tsconfig.lib.json` if placed in new directories
- Breaking changes are marked with `!` or `BREAKING CHANGE:` in the commit
- No `any` in production code (warning level -- justify if necessary)
- Types are defined in `types.ts`, not inline

### Code Ownership

`@project-kessel/hcc-framework` owns the entire repository. `.github/` and `CODEOWNERS` changes require admin review.

### Release Flow

Merges to `master` trigger automatic release: version bump from commit types, CHANGELOG update, git tag, npm publish, GitHub release creation. Demo packages (`demo-*`) are excluded from releases.
